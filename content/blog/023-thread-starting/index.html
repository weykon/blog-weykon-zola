从rust中认识线程编程

对线程的初步理解

在使用rust编程线程内容时，

Sender, Receiver 是发送和接收的端点 Sender是用send()发送, Receiver使用recv()接收。 tx 和 rx, "transmitter" 和 "receiver" 的缩写。

在使用多线程编程时，你希望有一种方式可以在线程之间发送数据，并希望有一种方式可以在另一个线程中接收数据。

Sender 和 Receiver 类型提供了这种能力。

比如如下代码实例解析：

// job for what ? -> 将一个元组Job实例 "Job(path.to_path_buf(), job_tx)" 发送到线程开始处理
// 在对于job_rx, 在线程代码中编写了job_rx的行为
// 那么从代码顺序的角度来看，简单来说，先声明在线程的处理过程，然后在job_tx这里激活运行("send()")并附带了参数
fn some_exec(){
    let (job_tx, job_rx) = crossbeam_channel::unbounded::<Job>();
    let (result_tx, result_rx) = crossbeam_channel::unbounded::<ProjectDir>();
    
    (0..num_threads)
        .map(|_| (job_rx.clone(), result_tx.clone()))
        .for_each(|(job_rx, result_tx)| {
            std::thread::spawn(move || {
                job_rx
                    .into_iter()
                    .for_each(|job| 
                        find_cargo_projects_task(
                            job, result_tx.clone(), verbose
                        )
                    )
            });
        });
    
    job_tx
        .clone()
        .send(Job(path.to_path_buf(), job_tx))
        .unwrap();
    
    result_rx
}

使用了crossbeam_channel::unbounded，是为主程序和线程之间创建了一对接发对象作为一条无界通道。
job_tx, job_rx 具备让线程能够取得主程序中闭包函数的作用域的数据。
也就是说，使用无界通道，在线程内部的任务中接收主线程发送的消息，从而获取闭包所在的作用域内的变量。

再看另一段代码

fn some_exec () {
    let (sender, receiver) = mpsc::channel();
    let mut workers = Vec::with_capacity(size);
    let receiver = Arc::new(Mutex::new(receiver));
    for id in 0..size {
        workers.push(Worker::new(id, Arc::clone(&receiver)));
    }
}

impl Worker {
    pub fn new(id: usize, receiver: Arc<Mutex<mpsc::Receiver<Message>>>) -> Worker {
        let thread = thread::spawn(move || loop {
            let message = receiver.lock().unwrap().recv().unwrap();

            match message {
                Message::NewJob(job) => {
                    println!("Woker {} got a job; executing.", id);
                    job();
                }
                Message::Terminate => {
                    println!("Worker {} was told to terminate", id);
                    break;
                }
            }
        });

        Worker {
            id,
            thread: Some(thread),
        }
    }
}

mpsc::channel 函数会创建一个多生产者单消费者 (multiple producer, single consumer) 的通道，
即可以有多个发送者向通道中发送消息，但是只能有一个接收者从通道中接收消息。
即说的是，在主程序有多个发送参数数据的实例，而在线程内部只有一个

先复习一下这个案例

use std::sync::mpsc;
use std::thread;
// mpsc : multiple producer, single consumer. 
pub fn main() {
    // a transmitter and a receiver
    let (tx, rx) = mpsc::channel();
    
    thread::spawn(move || {
        let val = String::from("hi");  // 线程中创建的String用tx发给tr
        tx.send(val).unwrap();
    });

    let received = rx.recv().unwrap(); // ，然后在主程接收
    println!("Got: {}", received);
}

从有界、无界看世界

首先，我不明白有界，是为何，我顺着理清楚
mpsc::channel() 的运作是创建一个队列数据，队列是有大小的，这种形式下不是无限的。
所以有界，这种情况会在什么需求下造成什么现象呢？
如果队列的长度超过了上限，则发送者会被阻塞，直到通道中有消息被接收。
这样可以避免内存泄漏，同时也可以降低内存分配的次数。
比如，在我们的代码中设计一段有界通道，以下：

    let (tx, rx) = crossbeam_channel::bounded(100);   // bounded 有界，unbounded 是无界
    // 在发送者线程中发送数据
    for i in 0..200 {
        tx.send(i).unwrap();
    }
    // 在接收者线程中接收数据
    for _ in 0..200 {
        let x = rx.recv().unwrap();
        // 在接收者线程中处理数据
        // ...
    }

这里设定了队列是100个，就是100个的容量。
在tx发送200个，但是只有100容量，当前100个在已经写入等待接受的时候，还没开始拿走，
他就会形成停止，
在处理完，即有去接收第1个tx, 就会腾出一个空位置，这时候，tx继续写入第101个
不断地交替运行。

这种情况造成的后果大概是，如果执行代码中一直没去接收数据，在超出数组范围的情况下，
就无法继续进行写入操作了
形式：在我的线程中要处理接收的任务，这个任务有一定的耗时，线程这里一时间处理不完在接收过程中的任务，
正好现在他的数组也满了，这个时候就写入不进去了，写入的进程就会卡住。
发送者一般是主线程，如果阻塞了，岂不是主线程卡住了，所以这样子不行。

无界

无界通道使用一种叫共享状态的数据结构

共享状态可以让多个线程同时访问，并且在多个线程同时修改时保证数据一致性

原理是使用互斥锁或者读写锁来实现的

内存陷阱

在使用无界通道，他的写入数据是由（开发）用户自己掌握了，
如果是一位无畏战士创建了大量的写入程序，会占用大量的内容，可能会造成内存泄漏。
（内存泄漏的简单来说就是程序分配的内存用完了，但却没有被释放。这样就会导致程序无法继续分配内存，从而导致程序崩溃或者性能下降。）

优美风景画

  • 互斥锁
    它能保证在任意时刻，只有一个线程访问共享状态。 Mutex 是 "Mutual Exclusion" 的缩写，lock().unwrap() 取得后，在函数结束会自动释放锁
use std::sync::{Arc, Mutex}; 
use std::thread; 

fn main() {

    let counter = Arc::new(Mutex::new(0));

    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            let mut num = counter.lock().unwrap();
            *num += 1;
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("Result: {}", *counter.lock().unwrap());

}
  • 读写锁
    能保证多个线程同时访问共享状态，但是在写入数据时会被阻塞。
use std::sync::{Arc, RwLock};
use std::thread;

fn main() {
    let counter = Arc::new(RwLock::new(0));

    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter);
        let handle = thread::spawn(move || {
            let mut num = counter.write().unwrap();
            *num += 1;
        });
        handles.push(handle);
    }

    for handle in handles {
        handle.join().unwrap();
    }

    println!("Result: {}", *counter.read().unwrap());
}

死锁

use std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let lock1 = Arc::new(Mutex::new(0));
    let lock2 = Arc::new(Mutex::new(0));

    let lock1_ref = lock1.clone();
    let lock2_ref = lock2.clone();

    thread::spawn(move || {
        // 锁住 lock1
        let mut num = lock1_ref.lock().unwrap();
        *num = 5;

        // 尝试锁住 lock2，这里就会发生死锁
        let mut num = lock2_ref.lock().unwrap();
    });

    thread::spawn(move || {
        // 锁住 lock2
        let mut num = lock2_ref.lock().unwrap();
        *num = 10;

        // 尝试锁住 lock1，这里也会发生死锁
        let mut num = lock1_ref.lock().unwrap();
    });
}
